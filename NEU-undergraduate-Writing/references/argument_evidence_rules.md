# 主张-证据写作规则

本文件用于把论文从“说明文字”变成“可审查的学术论证”。任何章节都不得先写结论再凭空补理由。

## 1. 主张-证据表

写作前为目标章节建立最小表格，内部使用即可，不必写入正文：

| 主张 | 证据 | 来源 | 状态 | 正文位置 |
|---|---|---|---|---|
| 本节要证明什么 | 用什么证明 | context.md / 代码 / 实验 / 文献 / 图表 | 已证实 / 待核验 / 待补 | 第几段 |

状态为“待核验”或“待补”的内容不得写成确定结论。

## 2. 可接受证据类型

- 文献证据：已核验的论文、标准、技术报告或权威资料。
- 项目证据：代码结构、提交记录、README、设计文档、配置文件。
- 实验证据：数据集、指标、主结果、消融、可视化、复杂度、失败案例。
- 理论证据：问题定义、公式推导、复杂度分析、假设条件。
- 工程证据：运行环境、系统流程、接口设计、部署或复现实验。

不要把“常识”“直觉”“模型认为”当作证据。

## 3. AXES 段落模型

每个核心段落优先使用 AXES：

| 部分 | 作用 | 写法 |
|---|---|---|
| A: Assertion 主张 | 段首提出本段观点 | “本文方法通过……缓解……” |
| X: eXample/Evidence 证据 | 给出数据、引用、代码事实或图表 | “表 X.Y 显示……” |
| E: Explanation 解释 | 解释证据为何支撑主张 | “这一结果说明……” |
| S: Significance 意义 | 回扣本节任务或引出下文 | “因此，该模块为……” |

不能每段都机械写成四句，但核心段落必须具备这四种功能。

## 4. 章节级证据要求

| 章节 | 主张类型 | 需要的证据 |
|---|---|---|
| 摘要 | 本文完成了什么、效果如何 | 方法关键词、验证方式、最关键结果 |
| 绪论 | 问题重要、现有方法不足、本文有贡献 | 背景引用、研究空白、贡献与证据预告 |
| 相关工作 | 某类方法的优势与不足 | 主题聚类、逐类比较、局限分析 |
| 方法 | 每个设计选择有必要 | 动机、输入输出、公式/流程、实现细节 |
| 实验 | 本文主张得到验证 | 主结果、消融、指标解释、对比基线 |
| 结论 | 本文回答了研究问题 | 对绪论贡献的回应、边界、未来工作 |

## 5. 实验-主张对应

实验章节中，每个实验前应能回答：

- 这个实验验证哪个主张？
- 使用什么数据、指标和基线？
- 读者应该观察哪个结果？
- 该结果支持、削弱还是限制了本文主张？

典型对应关系：

| 实验类型 | 验证对象 |
|---|---|
| 主结果对比 | 方法整体是否有效 |
| 消融实验 | 单个模块是否必要 |
| 参数敏感性 | 方法是否依赖特定超参数 |
| 可视化分析 | 方法是否产生预期中间行为 |
| 复杂度分析 | 性能收益是否值得计算代价 |
| 失败案例 | 方法边界是否被诚实说明 |

## 6. 禁止的论证捷径

- “实验结果表明本文方法有效”但不写哪个实验、哪个指标。
- “现有方法存在不足”但不给具体不足和引用。
- “本文具有创新性”但不说明相对谁、创新在哪里。
- “显著提升”“大幅改善”但没有数字或统计依据。
- “本文方法鲁棒性强”但没有跨数据集、噪声、消融或稳定性证据。

## 7. 证据不足时的正文写法

- 用 `【待补证】` 标记无法核查的断言。
- 用 `[CITE:关键词]` 标记需要文献支撑的位置。
- 用“在本文实验设置下”“初步结果显示”“从已有结果看”降低确定性。
- 直接说明缺口：“该结论仍需在更多数据集上验证。”
